# ADR 017: REVISED — LR2021 SPI Protocol Clarification

## Status
**SUPERSEDED by ADR-020** (2026-07-23)

This ADR was originally written with incorrect diagnosis. The original content
claimed the chip was NOT an SX1280 and mandated RadioLib. That was wrong.

## What Was Wrong With The Original ADR-017

1. **Diagnosis error**: A diagnostic using SX1280 1-byte opcodes (0xC0) was sent
   to an LR2021 chip. The chip returned garbage (0x21) because it doesn't
   understand SX1280 protocol. This was misinterpreted as "chip responds to SX1280."

2. **Wrong recommendation**: Mandated RadioLib LR2021 driver, which has NEVER
   worked on our hardware due to protocol mismatch (RadioLib uses 24-bit register
   addressing, our chip uses 2-byte opcodes).

3. **Premature ban**: Banned "SX1280 from all source" — but our raw SPI code
   never used SX1280 commands in the first place. It used correct LR2021 opcodes.

## Correct Understanding

- The chip IS an LR2021 (confirmed: module label "LORA2021-915", FCC ID "2RAD6S-LORA2021-915")
- It uses a **2-byte big-endian opcode SPI protocol** (documented in ADR-020)
- RadioLib's LR2021 driver does NOT work (different protocol: 24-bit register addressing)
- Our raw SPI code using 2-byte opcodes WORKS (1377 kbps verified)
- See **ADR-020** for the authoritative protocol specification and library decision

## What To Do Now

1. Follow ADR-020 for all LR2021 communication
2. Use the `LR2021Raw` library or raw 2-byte opcode SPI calls
3. Ignore any guidance from the original ADR-017 text
4. The `docs/lr2021-spi-protocol-reference.md` is the authoritative protocol doc
