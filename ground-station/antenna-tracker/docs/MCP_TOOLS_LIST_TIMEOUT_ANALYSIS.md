# MCP Tools List Timeout – Root Cause Analysis

## Symptom

When running:

```
bash scripts/dev_serve_and_call.sh
```

The output stops at:

```
--- Listing tools ---
Error: MCP error -32001: Request timed out
```

The server builds and starts correctly, but `tools/list` does not return any tools.

---

## Observations

- ✅ The server compiles successfully.
- ✅ The gateway starts successfully.
- ✅ The `initialize` lifecycle step completes.
- ❌ `tools/list` times out.

There are no lifecycle errors such as:

```
expect initialized request
```

This means the MCP handshake is working.

---

## Root Cause

In `rmcp` 1.x, tool definitions are **not automatically exported** merely by using `#[tool_router]`.

The server must explicitly attach the generated tool definitions to the `InitializeResult`.

Previously, the server advertised tool capability via:

```rust
ServerCapabilities::builder().enable_tools().build()
```

However, it did **not** include:

```rust
result.tools = Some(Self::tools());
```

As a result:

- The client believed the server supported tools.
- The server provided no tool definitions.
- `tools/list` never returned anything.
- The client timed out.

---

## Fix

Inside the `initialize()` method:

```rust
result.tools = Some(Self::tools());
```

This explicitly exports the tool metadata generated by the `#[tool_router]` macro.

---

## Why This Works

`#[tool_router]` generates a static `tools()` method.

In rmcp 1.x:

- Tool definitions are not auto-injected.
- They must be explicitly attached to the initialize result.

Once attached, `tools/list` returns:

- `move_antenna`
- `get_position`

---

## Status

After applying this fix, the server correctly exports tools and `tools/list` no longer times out.
